System Instruction: Execute Stage 5 (Secure Blazor WASM Frontend Foundation)

Context: We are building a unified, role-based Web Portal (AgriApp.Web) for our existing Clean Architecture .NET 8 Web API. The frontend must be responsive (mobile-first) using MudBlazor.

CRITICAL: Shared Models & API Contract

You MUST add a project reference to AgriApp.Core.

Do NOT create duplicate view models or DTOs in the frontend. You must strictly use the existing DTOs, Enums, and Entities from AgriApp.Core (e.g., WorkOrderDto, WorkOrderStatus, WorkOrderType).

The API base URL must be configured in Program.cs and injected into the typed HTTP clients. Assume standard REST routes (/api/auth, /api/workorders, /api/invoices, /api/attendance).

CRITICAL: Security & Validation Standards

Data Validation: Every form MUST use Blazor's <EditForm>, bound to a strongly-typed model from AgriApp.Core, and include <DataAnnotationsValidator />.

XSS Prevention: Strictly rely on Blazor's native data binding (@bind-Value). Do NOT use MarkupString to render any user-provided text.

API Security: All state-changing operations (POST/PUT/PATCH/DELETE) must gracefully handle 400 Bad Request and 403 Forbidden responses, parsing the API's validation errors and displaying them to the user via MudBlazor ISnackbar.

Step 1: Project Setup & Authentication Infrastructure

Create AgriApp.Web (Blazor WebAssembly Standalone). Install MudBlazor and Blazored.LocalStorage.

Create JwtAuthenticationStateProvider.cs. It must read the JWT from local storage, decode the claims (Role, CenterId), and manage the auth state securely.

Create JwtAuthorizationMessageHandler.cs (inherits DelegatingHandler) to attach the Bearer token to all outgoing HttpClient requests. Register this in Program.cs.

Step 2: Service Layer Generation (The API Wrappers)
Create an HttpServices folder. Generate typed HTTP clients that use the configured HttpClient and the JwtAuthorizationMessageHandler:

IAuthService: Handles /api/auth/login. On success, stores token in Blazored.LocalStorage and notifies the AuthStateProvider.

IWorkOrderService: Wraps CRUD operations for /api/workorders.

IAttendanceService: Wraps /api/attendance/clockin and clockout.

Step 3: Responsive UI & Role-Based Layout

Setup MainLayout.razor using MudBlazor (MudLayout, MudAppBar, MudDrawer).

Implement the Nav Menu using strictly bound <AuthorizeView> roles:

"Clock In/Out" (Roles: Staff, Sales, Supervisor).

"CRM / Inquiries" (Roles: Sales, Manager, SuperUser).

"Capacity Calendar" (Roles: Supervisor, Manager, SuperUser).

"Financials & Audit" (Roles: Manager, SuperUser).

Step 4: Global Error Handling Component

Implement a global error handling mechanism (either a Custom DelegatingHandler or a cascaded component) that intercepts HttpResponseMessage failures. If the API returns validation errors, display them cleanly using ISnackbar.Add(message, Severity.Error).

Decision Gate: Do NOT attempt to build the complex CRUD UI pages yet. Build the core infrastructure, the Auth State Provider, the HTTP Interceptors, and the Main Layout. Reply with "Secure Frontend Foundation Built" when complete.
